Progressive disclosure: load only what is needed, when needed

Principle

A skill exposes just its name and description up front; the full instructions and scripts load only when it is actually used.

WHY

Goldy follows the same idea internally. The report shows a one-line title and chips first, and tucks the command, result and references into drawers. The reader controls depth instead of drowning in everything at once.

Claude Code session transcripts (JSONL logs)nested explainer

Where the trace lives

Claude Code writes every session to a JSON-Lines file under:

~/.claude/projects/<slug>/<session-id>.jsonl

The slug is the project's absolute path with / and . replaced by -. Each line is one event object with a type:

  • user and assistant: message turns; message.content is an array of blocks (text, thinking, tool_use, tool_result).
  • file-history-snapshot, mode, permission-mode: session bookkeeping.

Because it is append-only JSONL, Goldy can parse it line by line without loading the whole session into memory, and a crashed write only loses the last line.

Networking: how this report reaches the web

Networking

Goldy is itself a networked tool. A few concepts make its links precise and durable:

  • URL anatomy: every reference is a URL with a scheme (https), a host (docs.python.org), a path (/3/library/os.path.html), an optional query string (explainshell uses ?cmd=...) and a fragment (#os.path.expanduser, which jumps to the exact definition).
  • HTTP requests: the research phase makes HTTP GET requests through WebFetch and WebSearch. When a site answered 403 Forbidden (a client-error status code), Goldy fell back to indexing it from search instead.
  • API versioning: doc links pin a version (docs.python.org/3/, Godot's /en/stable/) so they keep resolving to the right reference as upstream evolves.
  • HTTPS and TLS: links use https, so the connection to each source is encrypted and the server's identity is verified.
WHY

Reading a URL's parts, knowing what a status code means, and pinning an API version are what let Goldy deep-link to an exact, versioned, encrypted source rather than a fragile guess. The same literacy helps a developer debug any request, in any tool.

A URL is made of a scheme (https), an authority (host and optional port), a path, an optional query string after ?, and an optional fragment after #. HTTP status codes are grouped: 2xx success, 3xx redirection, 4xx client errors, 5xx server errors. A 403 means the server understood the request but refuses to authorize it, so a crawler must adapt rather than retry blindly.

How that script reads the file is a performance choice worth noting.

Stream the transcript line by line, not all at once

Performance

The parser reads the JSONL file one line per iteration instead of loading and parsing the whole session into memory.

WHY

A long session can be large. Iterating the file object yields one line at a time, so memory stays roughly constant and the work is a single O(n) pass. Tool results are indexed once into a dict for O(1) pairing with their tool calls.

Other paths considered so you can debug or adapt this
  • json.load the whole file

    Simplest to write, but JSONL is not one JSON document, and it would hold the entire session in memory at once.

  • Read all lines into a list first

    Convenient random access, but defeats the streaming win on large transcripts.

Reading files lazily in Pythontutorialrealpython.com

Why iterating a file object streams it rather than buffering the whole thing.

Read summary

Iterators produce values one at a time and only when asked, so they let you process data that is large or even unbounded without holding it all in memory. A Python file object is itself an iterator over lines, so looping directly over it reads one line per step. That is exactly how the parser walks a transcript without loading the whole session.

Time complexity (Big O)referenceen.wikipedia.org

Framing the single-pass parse and the O(1) dict lookups.

Read summary

Big O describes how an algorithm's running time grows with input size. A single pass over n lines is O(n), and a hash-map (dict) lookup is on average O(1) regardless of size. Indexing tool results into a dict once and then pairing them by id keeps the parser linear instead of quadratic.
